# FactSet MCP integration with Google Gemini Enterprise

This directory provides the minimal setup required to register a FactSet-connected Agent Development Kit (ADK) Assistant to Vertex AI Reasoning Engine, test it locally via the Python SDK, and properly inject FactSet OAuth tokens via Gemini Enterprise and Vertex Search.

## Architecture & Authentication Flow
The system demonstrates *Runtime User-Delegated Pass-Through Registration:*
1. **At Deployment:** The ADK fetches a single, temporary OAuth token locally to extract the MCP schema (`agent_engine.py`). It sanitizes and flattens this schema for strict Vertex AI typing logic (`agent.py`).
2. **At Runtime:** The user passes their token dynamically through Gemini Enterprise. As the session requests hit the Reasoning Engine, the token is extracted from the `readonly_context` state and appended as the `Authorization: Bearer <token>` in the MCP toolset headers.

## Prerequisites
- A Google Cloud Project with the `aiplatform.googleapis.com` and `discoveryengine.googleapis.com` APIs enabled.
- `gcloud` authenticated (`gcloud auth login` & `gcloud auth application-default login`).
- A FactSet Confidential Client ID, Secret, and Certificate (`config.json`).
- Python 3.10+ virtual environment (`.venv`).

The `agent_engine.py` script packages the underlying agent code (`agent.py`) and dependencies, pushing them to a managed execution Reasoning Engine in your GCP Region.

**Instructions:**
1. Open `agent_engine.py`.
2. Update the `TODO` placeholders with your `PROJECT_ID`, `LOCATION`, and `STAGING_BUCKET`.
3. Run the script:
   ```bash
   python agent_engine.py
   ```
4. *Save the newly created Reasoning Engine Resource ID provided in the console output.* Example format:
   `projects/<PROJECT_NUMBER>/locations/us-central1/reasoningEngines/<ENGINE_UID>`

## Step 3. Connect to Gemini Enterprise
The provided shell script automates the creation of the Server-Side OAuth2 resource in Discovery Engine, and the linking of the Reasoning Engine to Gemini Enterprise's Agent Builder.

**Instructions:**
1. Open `deploy_gemini_enterprise.sh`.
2. Update the variables under the `VARIABLES` block with your `PROJECT_ID`, `PROJECT_NUMBER`, and `AS_APP` (Gemini Enterprise App ID). Ensure your FactSet `CLIENT_ID` and `CLIENT_SECRET` are inserted.
3. Paste the `REASONING_ENGINE_RES` obtained from Step 1.
4. Run the configuration deployment:
   ```bash
   bash deploy_gemini_enterprise.sh
   ```
5. Your Agent will now appear in your Gemini Enterprise application and automatically enforce FactSet authorization upon invoking tools.

## 🛠 Troubleshooting: The Infinite Auth Loop
If you deploy the agent but get constantly kicked back to the login screen without completing the FactSet authorization:

1. **Missing Query Parameters (OAUTH_AUTH_URI)**: 
   Vertex AI Search processes the `authorizationUri` using strict JSON parsing when sending it to the API. If you embed variables containing ampersands (`&scope=mcp&response_type=code`) directly into the raw JSON payload body, Bash will not parse them reliably, causing Vertex to aggressively strip the `response_type` and `scope` parameters from the URI and default to a pure `?client_id=...&state=...` URL. 
   - **Fix:** Always build the full URI (`OAUTH_AUTH_URI`) as its own Bash variable first, and insert it safely into the JSON as `'${OAUTH_AUTH_URI}'` (which `deploy_gemini_enterprise.sh` handles out of the box).
   
2. **Invalid Client Secret**: 
   FactSet successfully redirects back to Vertex after login, but Vertex attempts a back-channel token exchange using the provided `CLIENT_SECRET`. If the secret is invalid, FactSet throws an error and Vertex silently drops the session, returning you to the browser prompt.
   - **Gotcha**: Do *not* mistake the base64 `state` parameter generated by Vertex Search (seen in the browser URL after returning from FactSet) as your FactSet `Client Secret`. The real secret will stringently come from the FactSet developer console alongside your `client_id`.

## 🚀 Architectural Best Practices & Crucial Findings

### 1. Extracting the Auth Token from Gemini Enterprise (Undocumented)
Public ADK documentation suggests that Gemini Enterprise injects tokens using the `auth_token` attribute on Context objects, or into the state dictionary prepended with `temp:` (e.g. `temp:factsetauth`). **This is incorrect for Reasoning Engine deployments.**
Gemini Enterprise dynamically injects the decoded JWT `Bearer` string directly into the Reasoning Engine's `readonly_context.state` dictionary utilizing **your literal Auth Resource ID** as the key.
- **The Fix:** Do not parse for `temp:`. Simply scan the `readonly_context.state.items()` dictionary for keys containing your prefix (e.g. `factset-auth`) and grab the corresponding string value. See `get_factset_headers` in `agent.py` for the robust implementation.

### 2. Handling API Drops Gracefully (Avoiding 500 Stream Errors)
If you provide an empty JSON header payload or a malformed JWT to the FactSet MCP server during a user's prompt sequence, FactSet's API gateways will aggressively terminate the TCP connection (`Connection Closed`) rather than replying with an HTTP 401. If your `agent.py` toolset builder doesn't catch this error, the ADK Python container natively crashes, returning a generic `500 Server Error` back to your UI.
- **The Fix:** Always wrap the `session.create_session()` and `session.list_tools()` routines in a `try...except` block in your `patched_get_tools`. Upon catching an exception, simply return `[]` (an empty array of tools). This allows the Gemini LLM to gracefully tell the user "I'm having trouble connecting to my FactSet tools right now" without actively destroying the web socket stream.

### 3. Enabling OpenTelemetry in Google Cloud Trace
Even if you pass `enable_tracing=True` via `AdkApp`, you will not see visual trace timelines in the Google Cloud Console for the FactSet API integration unless the underlying Vertex container is explicitly commanded to capture them.
- **The Fix:** When packing your container in `agent_engine.py`, explicitly define the telemetry environment variable:
  ```python
  agent_config = {
      ...
      "env_vars": {"GOOGLE_CLOUD_AGENT_ENGINE_ENABLE_TELEMETRY": "true"},
  }
  ```

### 4. Real-Time Timeline Grounding Tools
A major pitfall of Generative AI agents mapping to strict financial APIs (like `FactSet_GlobalPrices`) is that standard System Prompts do not know "today's exact date," causing severe hallucination in relative requests like "What did Apple do yesterday?"
- **The Fix:** Directly provide custom execution tools alongside your `McpToolset`. Create a standard Python function like `get_current_datetime()` and instruct the LLM in your system prompt to *always execute this function first*. This establishes the timezone math mathematically before the LLM attempts to populate `startDate` parameters.
